## 三、核心模块说明

### 模块 1：SKILL.md 技能文件

每个技能是一个独立文件夹下的 `SKILL.md`，结构分**两段**，中间用 `---` 分隔：

```
---                               ← 第一段开始（frontmatter 元数据）
name: inventory-analysis          ← 技能唯一名称
description: 分析仓库库存状况...  ← ★ 路由的关键，LLM 读这行决定是否调用
triggers: [库存, 缺货, 补货...]   ← 触发关键词（辅助说明）
negative: [采购单, 审批, 合同]    ← 排除关键词（防止误匹配）
---                               ← 第一段结束

# 库存分析技能                    ← 第二段（完整指令，匹配前不加载）
## 你的角色
## 分析流程
## 输出格式（必须遵守）
```

**为什么要分两段？—— 渐进式披露**

| 时机 | 加载内容 | Token 消耗 |
|------|---------|-----------|
| 启动时 | 只读 description（1行描述） | 极少 |
| 匹配后 | 加载完整指令（全文） | 按需 |

如果有 50 个技能，每次只加载 50 行描述做路由，而不是把 50 个完整技能全部塞给 Claude，既节省 Token，又避免上下文污染。这是本架构最核心的设计思想。

---

### 模块 2：技能路由（agent.py — match_skill）

```
用户输入
    │
    ▼
构建技能目录（仅描述，不含完整指令）
    │
    ▼
调用 Claude Haiku（轻量、便宜）
    │  系统提示："你是技能路由器，选择最匹配的技能名称，只输出名称"
    ▼
返回技能名称（如 "inventory-analysis"）
    │
    ▼
按需加载该技能的完整 SKILL.md 正文
```

**传统做法 vs SKILL.md 做法对比：**

| 传统做法 | SKILL.md 做法 |
|---------|--------------|
| 用关键词规则匹配（`if "库存" in text`） | LLM 读描述自主判断 |
| 规则多了维护困难 | 加技能只需新建文件夹 |
| 无法理解语义 | 能理解"盘点一下仓库"≈"库存分析" |
| 用大模型做路由，贵 | 用 Haiku（小模型）路由，省钱 |

**扩展方式**：在 `skills/` 下新建文件夹和 SKILL.md，路由自动发现，无需修改 `agent.py` 任何代码。

### 模块 3：MCP Server（mcp_server.py）

**MCP 是什么？**

MCP（Model Context Protocol）是 Anthropic 制定的开放标准协议，专门解决：**AI 模型如何以统一方式调用外部工具**。

- 没有 MCP：每个工具单独写对接代码，每换一个 AI 模型就要重写
- 有了 MCP：工具写一次，所有支持 MCP 的 AI 模型都能调用

**Server 结构：**

```python
@app.list_tools()       # ← 告诉 Claude "我有哪些工具"
async def list_tools():
    return [工具A, 工具B, 工具C]

@app.call_tool()        # ← Claude 要调工具时，执行这里
async def call_tool(name, arguments):
    if name == "query_inventory_summary":
        # 查 MySQL，返回结果
```

**本 Demo 提供的工具：**

| 工具名 | 功能 | 返回内容 |
|--------|------|---------|
| `query_inventory_summary` | 库存总览（聚合，永不返回明细） | 各分类物料数、总量、低库存统计 |
| `query_category_detail` | 分类明细（分层下钻，带 limit） | 指定分类的物料列表，最多30条 |
| `fuzzy_search_item` | 多字段模糊搜索 | 同时搜索名称/俗称/规格/别名，最多8条候选 |
| `query_low_stock_items` | 低库存预警明细 | 低于安全库存的物料，按紧急程度排序 |
| `get_item_by_code` | 按编号精确查询 | 单条物料完整信息 |

**为什么用 stdio 通信？**

```
agent.py（父进程）
    │  启动子进程
    ▼
mcp_server.py（子进程）
    ├── 父进程写入请求 → 子进程从 stdin 读取
    └── 子进程写入结果 → 父进程从 stdout 读取
```

| 通信方式 | 特点 | 适用场景 |
|---------|------|---------|
| **stdio（本 Demo）** | 进程内通信，零网络开销，延迟极低 | 内网部署、单机运行 |
| HTTP + SSE | 网络通信，支持远程调用 | 分布式、多人共享 |

内网场景选 stdio：不需要开端口，不需要网络配置，直接跑。

### 模块 4：Agent 工具调用循环（agent.py — run_agent）

关键是理解 **`stop_reason`** 这个信号——Claude 用它告诉 agent.py 下一步该做什么。

**完整循环示意：**

```
① 你问："有哪些物料低于安全库存？"
        │
        ▼
② Claude 收到：问题 + 技能指令 + 工具列表
   Claude 回复：stop_reason = "tool_use"
   （意思是："我需要调工具，先别结束"）
        │
        ▼
③ agent.py 看到 tool_use，执行工具：
   → session.call_tool("query_inventory_summary")
   → MCP Server 查 MySQL，返回数据
        │
        ▼
④ 把工具结果塞回消息历史，再次调用 Claude
   Claude 回复：stop_reason = "end_turn"
   （意思是："我拿到数据了，给你最终答案"）
        │
        ▼
⑤ 输出结构化分析报告
```

**两个关键信号：**

| stop_reason | 含义 | agent.py 的动作 |
|------------|------|----------------|
| `"tool_use"` | Claude 要调工具，还没结束 | 执行工具 → 把结果还给 Claude → 继续循环 |
| `"end_turn"` | Claude 给出最终回答 | 提取文本，结束循环，输出给用户 |

**为什么需要"循环"？**

一次复杂问题可能需要多轮工具调用，Claude 自己决定调几次、调哪个：

```
第1轮：调用 query_inventory_summary（看总体）
第2轮：调用 query_low_stock_items（看明细）
第3轮：调用 search_item("不锈钢")（看具体物料）
第4轮：stop_reason = end_turn → 给出综合报告
```

agent.py 只负责"执行工具并把结果还回去"，推理决策完全由 Claude 主导。

## 七、对接真实系统的关键设计方法

> 本章基于对湛江钢铁 PSCS 系统《物料代码管理设计说明书》的分析，
> 针对真实系统的两个核心挑战，提出可落地的方法论。

---

### 7.1 用户含糊描述 → 准确定位物料

**问题背景**

真实 PSCS 系统的物料查询逻辑是**物料代码完全匹配**（`CDM_ITEM_ID` 精确查询）。
但用户实际表达往往含糊，如："那个304的板"、"密封件"、"上次买的氮气"。

物料主数据包含多个文本字段，都是潜在的匹配入口：

| 字段 | 列名 | 长度 |
|------|------|------|
| 物料代码 | `CDM_ITEM_ID` | 20字符 |
| 长描述 | `CDM_DESCRIPTION` | 512字符 |
| 短描述 | `B_SHORT_DESCRIPTION` | 256字符 |
| 中文品名 | `B_CHINESE_PROD_NAME` | 512字符 |
| 型号规格 | `BS_MODEL` | 256字符 |
| 附加信息 | `BS_ADDITIONAL_INFORMATION` | 512字符 |

**方法：意图结构化 → 多字段模糊检索 → 候选确认**

分三步走，每步职责明确：

```
步骤一：Claude 做意图解析（调用 MCP Tool 之前）
         用户说"304不锈钢板"
         Claude 拆解 → { 关键词: ["304", "不锈钢", "板"], 推断类型: 原料 }

步骤二：MCP Tool 做宽松多字段检索
         SELECT * FROM CDM_INTERNAL_ITEM
         WHERE B_SHORT_DESCRIPTION   LIKE '%304%'
            OR B_CHINESE_PROD_NAME   LIKE '%不锈钢%'
            OR BS_MODEL              LIKE '%304%'
         LIMIT 10

步骤三：候选确认（Human-in-Loop）
         找到3条匹配，Claude 呈现列表，由用户指定：
         ① RAW-004 不锈钢板304 (2mm×1500mm) — 原料仓B
         ② RAW-011 不锈钢板304 (5mm×2000mm) — 原料仓C
         ③ AUX-008 不锈钢衬板304 — 辅料仓
```

**职责分工：**

| 角色 | 职责 |
|------|------|
| Claude | 语义理解，从自然语言中提取结构化查询条件 |
| MCP Tool / SQL | 宽松检索，返回候选集（不做最终判断） |
| 用户 | 从候选中确认目标（Human-in-Loop 最小单元） |

这与已拍板的 Human-in-Loop 原则完全一致——Claude 负责缩小范围，用户做最终裁定。

**扩展**：如果候选数量仍然很多（如 30 条），Claude 可以追加一轮追问："是哪个仓库的？大概什么规格？"，通过对话逐步收窄，而不是一次返回所有结果。

---

### 7.2 大量数据 → 与用户意图之间的平衡

**问题背景**

真实系统的库存表可能有数万条物料记录，直接查询返回大量数据：
- 超出 LLM 上下文窗口
- 用户不需要看原始行，只需要结论

分页（pagination）是数据库技术方案，但核心问题是**信息密度**——用户真正需要什么层级的答案。

**方法：分层查询 + 摘要优先 + 按需下钻**

不是把所有数据一次给出，而是按用户意图逐层展开：

```
第一层：聚合摘要（默认先返回这一层）
         "共 12,847 条物料，其中原料 4,312 条、备件 2,156 条
          低库存预警 47 项，其中原料类 23 项最多"

第二层：分类汇总（用户说"细化原料"）
         "原料中：钢板类 89条、线材类 45条、管材类 23条
          低库存：钢板类 12项 / 线材类 5项"

第三层：具体明细（用户说"看钢板类低库存"）
         "返回前20条，按库存占比升序排列..."

第四层：单条精确（用户给出物料代码）
         "RAW-004 不锈钢板304 详细信息..."
```

**MCP Tool 粒度设计对应上述四层：**

```python
# 工具一：只返回聚合，永不返回明细行
query_inventory_summary(category=None)

# 工具二：带过滤 + 限量的明细
query_inventory_detail(category, keyword, limit=20)

# 工具三：单条精确查询
get_item_by_code(item_code)
```

Claude 在 Agent 循环中自主决定调用顺序——先摘要，再按用户追问决定是否下钻，而不是一次倒出所有数据。

**工具设计原则：**

| 原则 | 说明 |
|------|------|
| 摘要优先 | 第一个工具永远只返回聚合，不返回明细行 |
| limit 由 Claude 控制 | 明细工具带 limit 参数，Claude 根据用户意图自行设定 |
| SQL 做聚合，AI 做推理 | 数量统计、均值计算在 SQL 层完成，AI 只做解释和建议 |
| 分层不分页 | 不是技术分页，而是按业务语义逐层下钻，由对话驱动 |

**与已有原则的关系**

本方法是 CONTEXT.md 中已拍板决策——"AI 只接收摘要数据，本地 SQL 预聚合"——在大数据量场景下的具体落地形式。摘要优先不是妥协，而是正确的信息架构。

---

### 7.3 两个方法的共同底座：MCP Tool 的粒度设计

无论是应对模糊描述还是大量数据，根本保障都来自 **MCP Tool 的粒度是否合理**：

```
粒度太粗（一个工具返回所有数据）→ 上下文爆炸，AI 无法处理
粒度太细（每个字段一个工具）→ 调用链过长，延迟高，逻辑碎片化
正确粒度 = 业务语义对齐：一个工具对应一个业务动作，返回量由 SQL 层控制
```

在从 Demo 迁移到真实 PSCS 系统时，工具粒度设计是最值得前期投入时间的环节。

### 测试1：大数据分层查询 ✅

**输入语句**：`库存整体情况，备件重点看`

**预期行为**：
1. 第1轮调用 `query_inventory_summary` 获取4行总览
2. 第2轮调用 `query_category_detail(category=备件, limit=15)` 显示15条，提示还有5条

**实际执行过程**：

```
[SKILL] 命中技能：inventory-analysis

第1轮工具调用（同一推理轮内连续调用两个工具）：
  ① query_inventory_summary（无参数）
     返回：共62种物料，12项低于安全库存
           辅料：14种  ⚠️4项低库存
           原料：16种  ⚠️3项低库存
           成品：12种  ⚠️3项低库存
           备件：20种  ⚠️2项低库存

  ② query_category_detail(category=备件, limit=15)
     返回：共20条，显示前15条（按库存紧急程度升序）
           SPA-002 液压缸密封件  ⚠️60%
           SPA-004 PLC模块S7-300 ⚠️67%
           ... 共15条 ...
           ─ 还有5条未显示，可继续缩小范围或指定物料编号查询 ─
```

**Agent 最终回答摘要**：
- 输出整体库存健康评估表（4行）
- 备件低库存预警：SPA-002（P1立即补货）、SPA-004（P2本周跟进）
- 补货建议量标注"建议值"
- 提示用户可查看备件剩余5条、切换其他分类、或查看全仓预警清单

**关键观察**：
- Claude 在**同一推理轮内**连续调用了 summary 和 category_detail，未等待用户确认，这是符合预期的高效行为（用户意图已明确）
- limit=15 由 Claude 自主设定，与工具默认值一致
- 分层下钻逻辑（摘要→明细→提示继续）完全按 SKILL.md 策略三执行

### 测试2：模糊描述 · 数字型关键词 ✅

**输入语句 A**：`304的库存情况怎么样`  
**输入语句 B**：`304`（裸数字，无上下文）

**预期行为**：调用 `fuzzy_search_item("304")`，在名称/规格/别名中命中不锈钢304相关物料

**实际执行过程（输入A）**：

```
[SKILL] 命中技能：inventory-analysis

第1轮工具调用：
  fuzzy_search_item(keyword='304')
  返回：1条候选
        RAW-004  不锈钢板304  [304 2.0×1500mm]  原料仓B
        库存:85.00吨  ⚠️低库存  命中:名称

Agent回答：展示候选表格，标注低库存预警，要求用户确认是否为目标物料
```

**实际执行过程（输入B，裸输入「304」）**：

```
[SKILL] 未命中任何技能（纯数字无触发词）→ 退到通用模式

第1轮工具调用：
  fuzzy_search_item(keyword='304')   ← 模型凭工具描述自主判断，调用正确
  返回：同上，1条候选

Agent回答：直接输出汇总表格，给出低库存预警，未要求确认
```

**两种输入的行为对比**：

| 维度 | 输入A「304的库存情况怎么样」 | 输入B「304」 |
|------|--------------------------|------------|
| SKILL 命中 | inventory-analysis ✓ | 未命中，通用模式 |
| 工具调用 | fuzzy_search_item('304') | fuzzy_search_item('304') |
| 物料命中 | RAW-004 ✓ | RAW-004 ✓ |
| 回答风格 | 严格候选确认流程，要求用户确认 | 直接给出结论，语气更主动 |

**关键观察**：
